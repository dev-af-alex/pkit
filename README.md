# pkit

Шаблонный репозиторий для полного цикла разработки: от продуктовой идеи до реализации через спецификацию, план и задачи. Используйте его как GitHub template для нового продукта или фичи.

## Что дает шаблон

- Единый и повторяемый процесс: идея -> конституция -> спецификация -> план -> задачи -> реализация.
- Прозрачные артефакты, которые удобно обсуждать, ревьюить и согласовывать.
- Автоматизированные навыки (skills) для генерации и проверки документов.
- Трассируемость решений: почему делаем -> что делаем -> как делаем -> чем именно заняты.

## Как начать (GitHub template)

1. На GitHub нажмите "Use this template" и создайте новый репозиторий.
2. Клонируйте его локально.
3. Откройте чат с агентом и запускайте навыки из `AGENTS.md` из корня репозитория.

## Подход и полный цикл разработки

Процесс разбит на независимые, проверяемые этапы. Каждый этап создает артефакт, который становится входом для следующего.

Идея
  -> product-intake (контекст)
  -> speckit.constitution (принципы проекта)
  -> speckit.specify (spec.md)
  -> speckit.clarify (уточнения)
  -> speckit.plan (plan.md + дизайн-артефакты)
  -> speckit.tasks (tasks.md)
  -> speckit.analyze (проверка согласованности)
  -> speckit.checklist (качество требований, опционально)
  -> speckit.implement (реализация по задачам)
  -> speckit.taskstoissues (issues в GitHub, опционально)

## Полный workflow по шагам

Ниже перечислены все шаги в правильном порядке. Каждый шаг - это навык, который запускается через вашего агента (см. `AGENTS.md`).

### 1) product-intake - сбор продуктового контекста

**Когда использовать:** в самом начале, чтобы собрать контекст и подготовить входы для конституции.

**Команда:**

- `product-intake <краткое описание продукта/идеи>`

**Что делает:**

- Создает workflow-документ (инструкция для следующего шага) и задачи.
- Фиксирует недостающие вопросы, если не хватает данных.

**Выход:**

- Workflow-шаблон с входными данными для конституции.
- Обновленный `tasks.md` для product intake.

**Дальше:** `speckit.constitution`.

---

### 2) speckit.constitution - конституция проекта

**Когда использовать:** после product-intake или когда нужно обновить принципы проекта.

**Команда:**

- `speckit.constitution <входные данные/принципы>`

**Что делает:**

- Заполняет `.specify/memory/constitution.md` конкретными принципами и правилами.
- Синхронизирует шаблоны спецификации, плана и задач с конституцией.

**Выход:**

- Обновленная `.specify/memory/constitution.md`.
- Отчет о синхронизации и версиях.

**Дальше:** `speckit.specify`.

---

### 3) speckit.specify - спецификация фичи

**Когда использовать:** чтобы получить `spec.md` из описания фичи или идеи.

**Команда:**

- `speckit.specify <описание фичи>`

**Что делает:**

- Создает ветку и директорию фичи.
- Формирует спецификацию `spec.md` по шаблону.
- Генерирует чеклист качества спецификации.

**Выход:**

- `spec.md` и `checklists/requirements.md` в директории фичи.

**Дальше:** при необходимости `speckit.clarify`, затем `speckit.plan`.

---

### 4) speckit.clarify - уточнения спецификации

**Когда использовать:** если в `spec.md` есть неопределенности или спорные места.

**Команда:**

- `speckit.clarify` (или с контекстом в аргументах)

**Что делает:**

- Задает до 5 приоритетных вопросов.
- Записывает ответы в `spec.md` и в раздел Clarifications.

**Выход:**

- Обновленный `spec.md` с устраненными неопределенностями.

**Дальше:** `speckit.plan`.

---

### 5) speckit.plan - технический план

**Когда использовать:** когда спецификация готова и нужен план реализации.

**Команда:**

- `speckit.plan <контекст стека/требований>`

**Что делает:**

- Создает `plan.md` и артефакты дизайна (например `research.md`, `data-model.md`, `contracts/`, `quickstart.md`).
- Обновляет агентский контекст.

**Выход:**

- `plan.md` и связанные артефакты в директории фичи.

**Дальше:** `speckit.tasks`.

---

### 6) speckit.tasks - список задач

**Когда использовать:** после готового плана.

**Команда:**

- `speckit.tasks <контекст>`

**Что делает:**

- Генерирует `tasks.md` с фазами и зависимостями.
- Привязывает задачи к user stories.

**Выход:**

- `tasks.md` в директории фичи.

**Дальше:** `speckit.analyze` (рекомендуется), затем `speckit.checklist` (по необходимости), затем `speckit.implement`.

---

### 7) speckit.analyze - анализ согласованности

**Когда использовать:** после генерации `tasks.md`, чтобы проверить согласованность `spec.md`, `plan.md`, `tasks.md`.

**Команда:**

- `speckit.analyze`

**Что делает:**

- Делает read-only анализ и выдает отчет о несоответствиях, пробелах и рисках.

**Выход:**

- Отчет в ответе агента (файлы не меняются).

**Дальше:** при необходимости обновить документы, затем продолжить.

---

### 8) speckit.checklist - чеклисты качества требований

**Когда использовать:** если нужен отдельный чеклист качества требований (например безопасность, UX, API).

**Команда:**

- `speckit.checklist <область/фокус>`

**Что делает:**

- Создает новый чеклист в `checklists/` с проверками качества требований (не тесты реализации).

**Выход:**

- Новый файл чеклиста, например `checklists/ux.md` или `checklists/security.md`.

**Дальше:** завершить чеклисты перед `speckit.implement`.

---

### 9) speckit.implement - реализация по задачам

**Когда использовать:** когда `tasks.md` готов и чеклисты (если есть) закрыты.

**Команда:**

- `speckit.implement`

**Что делает:**

- Выполняет задачи по фазам.
- Отмечает выполненные задачи в `tasks.md`.
- Следит за зависимостями и чеклистами.

**Выход:**

- Изменения в коде и обновленный `tasks.md`.

---

### 10) speckit.taskstoissues - задачи в GitHub issues (опционально)

**Когда использовать:** если нужно перенести задачи в GitHub issues.

**Команда:**

- `speckit.taskstoissues`

**Что делает:**

- Создает GitHub issues из `tasks.md` (только если remote - GitHub).

**Выход:**

- Issues в GitHub репозитории.

## Быстрый сценарий "с нуля"

1. `product-intake <описание идеи>`
2. `speckit.constitution <принципы/контекст>`
3. `speckit.specify <описание фичи>`
4. `speckit.clarify` (если есть неопределенности)
5. `speckit.plan <контекст стека>`
6. `speckit.tasks`
7. `speckit.analyze` (рекомендуется)
8. `speckit.checklist <фокус>` (опционально)
9. `speckit.implement`
10. `speckit.taskstoissues` (опционально)

## Содержимое репозитория

- `README.md` - обзор и инструкции (этот файл).
- `AGENTS.md` - инструкции для агентов и список навыков.
- `LICENSE` - лицензия шаблона.
- `.codex/prompts/` - файлы навыков (skills).
- `.specify/` - скрипты, шаблоны и память процесса (создаются и используются навыками).

## Примечания

- Все команды предполагают запуск из корня репозитория.
- Навыки читают и пишут файлы в `.specify/` и в директорию конкретной фичи.
- Детали использования и список навыков см. в `AGENTS.md`.
